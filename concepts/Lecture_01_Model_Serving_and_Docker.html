Why most ML models never see real users

The "notebook → production" gap, in three numbers

A trained model is barely 10% of the work. The other 90% is making sure it runs reliably, repeatably, and at the speed real users expect — and that's where most academic projects quietly die.

// Industry estimate
~85%

// Latency budget
< 200 ms

// Engineering effort
3–4×
Building the serving + monitoring infrastructure often costs 3–4× more engineering hours than training the model itself. Tools like Docker, K8s and MLflow exist to shrink this number.

So what is model serving?

A definition, then the four jobs every server must do

Definition

Model serving is the process of taking a trained model artifact (a .pkl, .pt, .h5, or SavedModel) and exposing it behind a stable interface — usually HTTP/REST or gRPC — so that other software can request predictions in real time, at scale, and with confidence.

1
Load the model artifact into memory exactly once at startup, then keep it warm — model loading is slow, predictions must be fast.
2
Predict on demand. Accept JSON / protobuf input, run preprocessing → model.predict() → postprocessing, and return a structured response.
3
Scale to many concurrent users. Multiple worker processes, GPU batching, autoscaling — all without changing the model itself.
4
Observe what's happening: log every request, track latency & errors, version the model, and roll back instantly when something goes wrong.

What makes ML deployment uniquely hard?

Three pain points that turn into technology choices

A trained model is not a normal "binary". It carries environment, data, and drift with it. Each of the three pain points below maps to a piece of tooling we'll meet in this and the next lab.

1Heavy & messy artifacts

Model weights are hundreds of MB to several GB. They depend on specific Python, CUDA, and library versions. Move them carelessly and your code crashes on the server.

2Stateful runtime

Unlike a CRUD service, ML servers must preload the model, manage GPU memory, and reuse it across thousands of requests — restarts are expensive.

3"Works on my notebook"

Tiny version drifts (numpy 1.26 vs 1.27) silently change predictions. Reproducibility between training and serving is a real, daily problem.

// The toolchain Docker freezes the environment. MLflow freezes the artifact + metadata. Kubernetes (next lab) freezes the runtime topology. Together they erase "works on my machine".

From model.pkl to a live HTTP service

Watch the same model walk through three forms

Take a moment to follow how a single trained model travels from a file on disk, to a web service, to a portable container. Each panel adds one capability — without changing the model itself.

// Step 1 · Train & save
A model on disk

In your notebook you fit RandomForestClassifier and dump it to model.pkl. The file knows nothing about how it will be used.

# train.py from sklearn.ensemble import RandomForestClassifier import joblib clf = RandomForestClassifier() clf.fit(X_train, y_train) joblib.dump(clf, "model.pkl") ✓ saved 4.2 MB to model.pkl
// Step 2 · Wrap in a server
A live HTTP endpoint

A tiny FastAPI app loads the pickle once on startup and exposes POST /predict. Now any service in your stack can call it.

# app.py from fastapi import FastAPI import joblib app = FastAPI() model = joblib.load("model.pkl") @app.post("/predict") def predict(payload: dict): y = model.predict([payload["x"]]) return {"y": y.tolist()}
// Step 3 · Containerise
A portable artifact

Wrap the app + model + Python deps in a Docker image. The same container now runs identically on a teammate's laptop, a CI runner, or a cloud node.

# Dockerfile FROM python:3.11-slim WORKDIR /app COPY requirements.txt . RUN pip install -r requirements.txt COPY . . EXPOSE 8000 CMD ["uvicorn", "app:app", "--host", "0.0.0.0"]
// Lesson Three stages, same model. Each layer adds portability without modifying the predictions. The container is the unit you'll deploy, scale, and roll back in production.

Meet Docker · your portable ML kitchen

Three words you must know: image, container, registry

Docker is the de-facto standard for packaging software. For ML, it solves a single, painful problem: "my code, my Python version, my CUDA, my model — all together, exactly the same, on every machine."

1
Image — a read-only recipe. Layered filesystem with the OS, Python, libraries, and your model. Built once, used forever. Lives in a registry as repo:tag.
2
Container — a running instance of an image. Isolated process tree, network, and filesystem. Start it, stop it, throw it away — the image is unchanged.
3
Registry — a marketplace for images. docker pull / push moves images between your laptop, CI, and the cloud. Docker Hub, GitHub Container Registry, ECR.
// Why ML loves Docker
Reproducibility, by construction
  • Pin Python, CUDA, and library versions in requirements.txt — once.
  • Same image hash means same predictions, byte-for-byte.
  • Run on a teammate's laptop, GitLab CI, or a Kubernetes pod with zero code changes.
  • One artifact replaces a 30-step "how to set up the project" wiki page.
// Mental model
Image is the class, container is the instance
Dockerfile → Image → Container

A Dockerfile is the source code. Run docker build to compile it into an image. Run docker run to launch one or more containers from it.

Containers vs Virtual Machines

Why a container starts in milliseconds and a VM in minutes

Both let you run software in isolation, but they isolate at very different layers. A VM virtualises the hardware and ships an entire OS. A container virtualises the process and shares the host kernel. For ML serving, this difference is the gap between seconds and hours of iteration time.

// Virtual Machine
Heavy isolation, full OS
  • Boot time: 30 s – several minutes
  • Disk size: typically 10 – 30 GB per VM
  • Resource overhead: an entire kernel + OS per VM
  • Strongest isolation — useful for multi-tenant clouds
// Container
Lightweight isolation, shared kernel
  • Start time: milliseconds to a few seconds
  • Disk size: typically 50 MB – 2 GB
  • Overhead: a single isolated process tree
  • Perfect for per-service ML deployments
// Why ML cares When iterating on a model, you may rebuild & restart the serving image dozens of times an hour. Containers make that loop cheap; VMs would make it unworkable. Same isolation guarantees you need, a fraction of the weight.

Anatomy of a model-serving Dockerfile

Every line earns its place — read top-to-bottom

A clean ML Dockerfile follows a predictable rhythm: pick base → set workdir → install deps → copy code → expose port → run server. The order matters: anything that changes often goes last so Docker's layer cache stays fast.

# 1) Base image — small, official, version-pinned FROM python:3.11-slim # 2) Working directory inside the container WORKDIR /app # 3) Install OS-level deps your wheels need (e.g. for numpy/scipy/lightgbm) RUN apt-get update && apt-get install -y --no-install-recommends \ build-essential libgomp1 && rm -rf /var/lib/apt/lists/* # 4) Copy ONLY the requirements first — caches the heavy pip install layer COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt # 5) Now copy the rest (model, code, configs) COPY app.py model.pkl ./ # 6) Document the network port the server listens on EXPOSE 8000 # 7) Sensible defaults — non-root user, predictable entrypoint USER 1000:1000 CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]
Layer caching · the one rule

Order your COPY instructions least → most volatile. requirements.txt rarely changes; your app.py changes every commit. Putting deps first means daily builds finish in seconds instead of minutes.

Image is a blueprint, container is an object

If you know class and new, you already know this

Take any OOP language. A class is a written-once template — fields, methods, defaults. An object is what you get when you new it. Multiple objects can come from the same class; each has its own state, but the class itself never changes. Docker is identical: one image (the blueprint), many independent containers (the instances). Real-life story · the day Docker saved a launch

A small Dhaka clinic, a TB X-ray classifier, and one disastrous Monday

A team of two final-year students at BRAC University trained a chest-X-ray model that flags tuberculosis with 97.4 % accuracy on the public Shenzhen + Montgomery datasets. They demoed it to a clinic in Mohammadpur. The clinic loved it and signed off on a 2-week deployment. Here's how week 1 actually went.

// Day 1 · Saturday
"It works on my laptop"

Tania's laptop has Python 3.11, OpenCV 4.9, PyTorch 2.3, CUDA 12.1, and libjpeg-turbo installed. Her notebook reads a chest X-ray, returns {"label":"tb","prob":0.972} in 380 ms. Demo day passes. The clinic asks: "Can it run on our box by Monday?"

# Tania's laptop · works perfectly $ python predict.py xray_42.png {"label": "tb", "prob": 0.972, "latency_ms": 380}
// Day 3 · Monday
The clinic's server bites back

The clinic's old Ubuntu 18.04 box has Python 3.6 and an ancient OpenCV. pip install takes 40 minutes — half the wheels won't compile. After hacking it together, the model returns {"label":"normal","prob":0.91} on the same X-ray. Wrong. The OpenCV version reads JPEGs differently → preprocessing differs → predictions differ. Two days lost.

# Clinic's server · same image, different answer $ python predict.py xray_42.png {"label": "normal", "prob": 0.91, "latency_ms": 2400} # 😱 a sick patient just got cleared
// Day 7 · Friday
One Dockerfile · same answers everywhere

Tania writes a 12-line Dockerfile pinning every dependency — Python 3.11, OpenCV 4.9, libjpeg-turbo, the model weights, even the timezone. docker build runs once. The same image now boots on her laptop, the clinic's server, and a tablet at the doctor's desk. Predictions match byte-for-byte in all three places.

$ docker run -p 8000:8000 ghcr.io/tania/tb-api:1.0 INFO: Application startup complete. $ curl -F file=@xray_42.png localhost:8000/predict {"label": "tb", "prob": 0.972, "latency_ms": 410} # ✓ identical on every machine
// The lesson A trained model is not a deployable artefact. The deployable artefact is the model + every system dependency, frozen together. Docker turns "works on my laptop" into "works everywhere, byte-for-byte" — and in healthcare or finance, that difference can be life-changing.

Smaller, safer, faster images

Five tricks that take an ML image from 1.8 GB to 240 MB

A naïve FROM python:3.11 + pip install can balloon to gigabytes. Each MB you save shrinks pull time, attack surface, and cluster cost. The five practices below cover >90% of production wins.

1
Use a slim or distroless base. python:3.11-slim is ~120 MB; the full python:3.11 is ~900 MB. For pure inference, gcr.io/distroless/python3 goes even lighter and ships no shell — much smaller attack surface.
2
Add a .dockerignore. Stop sending .git/, __pycache__/, datasets, and notebooks into the build context. Faster builds, no secrets leaked, smaller images.
3
Multi-stage builds. Compile heavy wheels in a builder stage, copy only the installed packages into a tiny final stage. Common 4–5× size reduction on ML images.
4
Pin everything & clean caches. Pin Python (3.11.9), pin libraries (scikit-learn==1.5.0), and end RUNs with rm -rf /var/lib/apt/lists/* && pip cache purge.
Multi-stage Dockerfile · template
# ----- stage 1 · builder (heavy) ----- FROM python:3.11-slim AS builder WORKDIR /app RUN apt-get update && apt-get install -y --no-install-recommends build-essential COPY requirements.txt . RUN pip install --user --no-cache-dir -r requirements.txt # ----- stage 2 · runtime (slim) ----- FROM python:3.11-slim WORKDIR /app COPY --from=builder /root/.local /root/.local # copy ONLY installed wheels COPY app.py model.pkl ./ ENV PATH=/root/.local/bin:$PATH EXPOSE 8000 HEALTHCHECK --interval=20s --timeout=3s CMD curl -f http://localhost:8000/health || exit 1 USER 1000:1000 CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]
// Bonus 5th Run as a non-root user (USER 1000:1000) and add a HEALTHCHECK. Both are free, both make orchestrators (Kubernetes!) much happier when they take over next session.

Build → Tag → Run

The container lifecycle, in three commands

Once your Dockerfile is right, the daily workflow collapses into three muscle-memory commands. Learn these and 80% of "DevOps for ML" feels routine.

// Stage 1
Build
Compile the Dockerfile into an image:
docker build -t iris-api:0.1 .
Docker reads each instruction, creates a cached layer, and stores the final image locally.
// Stage 2
Tag
Give the image a versioned, registry-aware name:
docker tag iris-api:0.1 ghcr.io/team/iris-api:0.1
Tags are how rollback works — keep :0.1, :0.2, :latest.
// Stage 3
Run
Launch a container from the image:
docker run -p 8000:8000 iris-api:0.1
The -p flag maps host port → container port. curl localhost:8000/predict now works.

When one container isn't enough · docker compose

Multi-service ML stacks in a single YAML file

A real ML service rarely runs alone — it talks to a cache, a database, a feature store, a tracking server. docker compose describes the whole stack as one declarative file and starts it with one command.

// Why compose
One up command, the whole environment
  • Defines multiple containers (services) and how they talk on a private network.
  • Manages volumes for model files, datasets, and logs that must outlive the container.
  • Maps ports, sets env vars, configures health checks.
  • Reproducibly spins up the same dev/test environment for every teammate with docker compose up.
  • Perfect local mirror of what Kubernetes will do tomorrow — same mental model, smaller scale.
// Sample stack
Predict API + Redis cache + MLflow
# docker-compose.yml services: api: build: . ports: ["8000:8000"] environment: - MODEL_PATH=/models/model.pkl - REDIS_URL=redis://cache:6379 depends_on: [cache, mlflow] cache: image: redis:7-alpine mlflow: image: ghcr.io/mlflow/mlflow:latest ports: ["5000:5000"] command: mlflow server --host 0.0.0.0
// One command Run docker compose up and Docker starts all three services, wires them on a private network, and tears them down with docker compose down. Identical experience for every team member.

MLflow · the open-source ML lifecycle toolbox

Track every run, log every artifact, ship the right model

MLflow answers four questions every team eventually asks: "What did we train? What did it score? Where is the artifact? Which version is in production?" It does so through four loosely-coupled components — pick what you need, ignore the rest.

1
Tracking — log parameters, metrics, and artifacts for every training run. mlflow.log_metric("auc", 0.93) writes to a queryable backend & UI.
2
Models — a standard packaging format. mlflow.sklearn.log_model(clf, "model") bundles weights + signature + Conda env in one folder.
3
Projects — declare how to run your code via an MLproject file. Anyone can mlflow run . and get the same experiment.
4
Registry — a versioned catalogue of models with stages: None → Staging → Production → Archived. The serving layer reads from here.
# Track an sklearn run + log the artefact in 6 lines import mlflow, mlflow.sklearn mlflow.set_experiment("iris-baseline") with mlflow.start_run(): mlflow.log_params({"n_estimators": 100, "max_depth": 6}) clf.fit(X_train, y_train) mlflow.log_metric("accuracy", clf.score(X_test, y_test)) mlflow.sklearn.log_model(clf, "model") # <-- the artefact your Docker image will load
